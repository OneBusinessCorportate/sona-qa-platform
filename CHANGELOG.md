# Журнал изменений

Запись формата «что / зачем / результат» обязательна для каждого PR, меняющего код: без неё робот `doc-sync` краснит и мердж заблокирован. Форточка для заведомо тривиальной правки — метка PR `skip-changelog`.

## 2026-07-22
- Ежедневный отчёт по тикетам Sona теперь считает закрывшийся (вчерашний) день, а не сегодняшний. Что: плановая отправка (`0 19 * * *` / GitHub Action в 15:00 UTC) и публичный эндпоинт `/api/cron/sona-tickets-daily` без параметра `?date` теперь берут дату по умолчанию через новый хелпер `yesterdayInTz` (предыдущий локальный день в `env.tz`), а не `todayInTz`; явный `?date=` по-прежнему приоритетен (ручная переотправка конкретного дня), интерактивные эндпоинты дашборда и кнопка «Отправить сейчас» продолжают показывать выбранную/сегодняшнюю дату. Зачем: Сона заносит проверки задним числом и поздно ночью — 5 проверок за 20.07 были внесены в 01:45–02:19 по Еревану уже 21.07, поэтому отчёт за 20.07 (отправлен в 19:00 20.07, до внесения) их не увидел, а отчёт за 21.07 искал `checking_date=21.07` и показал «Всего тикетов: 0»; проверки не попадали ни в один отчёт. Результат: отправка в 19:00 отчитывается за день, который к этому моменту уже полностью заполнен (`checking_date` вчера), поэтому поздние/задним числом внесённые проверки считаются ровно один раз; логика подсчёта (`buildSonaTicketsDaily`) не менялась, добавлены юнит-тесты `time.test.ts` (граница суток/месяца/года, локальный день vs UTC), 49 серверных тестов зелёные, комментарий GitHub Action приведён в соответствие с фактическим поведением.

## 2026-07-21
- Новый формат ежедневного Telegram-отчёта по тикетам Sona. Что: блок «По бухгалтерам» вместо построчного списка бухгалтеров теперь показывает две сводные строки — «Апелляций от бухгалтеров» (сколько проверок Соны бухгалтеры оспорили, `accountant_response_status = 'appealed'`) и «Проверок от Сони» (сколько апелляций Сона уже рассмотрела — принято + отклонено); удалён хвостовой блок «Проверка точности» с пустыми плейсхолдерами `___` и строкой «Подтвердите число в дашборде». Зачем: Сона просила убрать неинформативный блок с прочерками и вместо списка имён видеть агрегат по апелляциям. Результат: сообщение короче и по делу, счётчики берутся из уже собираемого `responses`-tally (логика подсчёта тикетов не менялась), тесты форматтера обновлены, 45 серверных тестов зелёные. `formatSonaTicketsDailyText` — единственное изменение; per-accountant разбивка (`byAccountant`) остаётся в JSON-ответе крона и на дашборде.
- Добавлены два клиента в справочник компаний (`public.mqa_chats`): B-4836 «YUNGLIN WANG LLC» (ХВХХ 20249358, бухгалтер Օլյա) и B-4827 «KIDSH LLC». Что: идемпотентная миграция `server/db/2026-07-21_add_missing_companies_yunglin_kidsh.sql` + вставка применена в проект «OB FAQ». Зачем: обеих компаний не было в общей таблице-справочнике, поэтому они не появлялись в выпадающем списке вкладки «Проверка» и Сона не могла их проверять. Результат: обе заведены со статусом Active и видны в списке компаний.

## 2026-07-20
- Опрос «что улучшить в платформе» для Соны. Что: после 3-й проверки за день (или вечером, если была хотя бы одна проверка) платформа один раз в день предлагает Соне короткий опрос (удобство 1–5, что замедляло работу, что улучшить); ответы пишутся в новую таблицу `sqa_platform_feedback` (RLS включён, без anon-политики — только service-role), добавлены эндпоинты `/api/platform-feedback` (`GET /prompt`, `GET /`, `POST /`), чистый модуль `survey.ts` (правило показа: порог 3 проверки / вечерний час) с юнит-тестами и модалка `PlatformSurveyGate` (App «бампает» её после каждой сохранённой проверки). Зачем: пункт задачи «после 3-й проверки / вечером спрашивать, что улучшить в платформе» — собрать обратную связь Соны по удобству. Результат: ненавязчивый опрос не чаще раза в день («Пропустить сегодня» и отправленный ответ прекращают показ на сегодня), 46 серверных тестов зелёные, клиент типизируется и собирается. Аддитивно — `sqa_*` only, ingest-контракт не тронут.
- Апелляции бухгалтеров видны и решаются в платформе Соны. Что: `/tickets/feed` и `/tickets/:id/feedback` теперь отдают апелляции из `kk_problem_appeals` (ключ `sona:<ticket>`), новый `POST /tickets/:id/appeal/resolve` выносит решение (принять/отклонить), а вкладки «Ответы бухгалтеров» и «Тикеты» показывают текст апелляции, статус и кнопки решения. Зачем: бухгалтер подаёт апелляцию по замечанию Соны в приложении бухгалтеров, но лента Соны читала только формы/комментарии/файлы — сами апелляции были не видны в её платформе (только в управленческой странице чужого приложения). Результат: решение здесь зеркалит `resolveAppeal` kk-приложения (одобрено → замечание снято `not_problematic` + штраф аннулирован; отклонено → замечание активно, возвращается бухгалтеру), трекер «Мои апелляции» у бухгалтера обновляется, проверяющий остаётся анонимным; чистая логика вынесена в `server/src/appeals.ts` с юнит-тестами.

## 2026-07-16
- «Ответы бухгалтеров»: добавлен переключатель «Показать закрытые» (`GET /tickets/feed?closed=1`) и исправлен текст пустого состояния. Зачем: лента скрывала обработанные тикеты (`explained_accepted`/`fixed`/`auto_resolved`), поэтому реально отправленные ответы бухгалтеров (формы + комментарии) были не видны Соне, а сообщение ложно гласило «ничего не отправляли». Результат: по умолчанию видны только открытые диалоги, но Сона может включить показ закрытых и увидеть все ответы; пустой текст теперь различает «нет открытых» и «ничего не отправляли».
