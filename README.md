# Sona QA Platform

Платформа ежедневной проверки **качества бухгалтерских услуг** (Сона).
Самостоятельный проект (не путать с QA-системой Маргариты по коммуникациям —
это про **бухучёт**). Построен по тем же паттернам, что и форма Маргариты
(`mqa_*`), но живёт в отдельных таблицах `sqa_*` и в отдельном приложении.

## Что внутри

UI намеренно минималистичный: только то, что нужно для ежедневной проверки —
без лишних кнопок и полей. Имя пользователя в интерфейсе не упоминается.

- **Вкладка «Проверка»** — ежедневная проверка качества по каждой компании:
  выбор компании из списка → авто-подтяжка ответственного бухгалтера. Указывается
  **проверяемый отчёт** (НДС / оборот / другое — в порядке приоритета),
  **степень риска**, при желании — период. Качество оценивается по
  **чек-листу из 9 пунктов** (Да/Нет — как в рабочем файле): просрочка, счета
  подписаны, корректность/полнота, подтверждение первичкой, формат/сдача, ошибки,
  камеральные требования, штрафы, внутренние стандарты. **Оценка %** считается
  автоматически = доля «хороших» ответов из 9 × 100 (напр. 8/9 = 88.9%).
  Один флажок **«Проблема»** превращает запись в **тикет** (+ флаг «🔴 Срочно»).
  В этой же форме можно указать **дату проверки** (по умолчанию сегодня — можно
  внести задним числом) и **доходы и расходы по компании** (сумма + раздел),
  чтобы видеть, какую сумму и в какой раздел бухгалтер включил.
- **Вкладка «Отчёты»** — три блока:
  - **Отчёт аудитора** — по каждому бухгалтеру **проверено / всего** и средняя
    оценка, плюс редактируемый **план на завтра** (сколько отчётов кому назначено);
  - **сводка за день** — проверки, средняя оценка, проблемы, тикеты;
  - **доходы и расходы по компаниям** — доход/расход/сальдо за день и разбивка
    по компаниям и разделам (из сумм, введённых в проверках);
  - **проверки за день** — список внесённых проверок с возможностью **удалить**
    ошибочную запись (связанный тикет удаляется вместе с ней).
  Все формируются автоматически из заполненных проверок.
- **Вкладка «Общая оценка»** — взвешенный скоринг бухгалтеров за период
  (журнал «Общая оценка» Соны). Пять критериев **К1 ошибки (×0.1), К2 сроки (×0.3),
  К3 отчётность (×0.2), К4 документы (×0.3), К5 доработки (×0.1)** сводятся в
  **Итог Q** (0–100) и **уровень** (премирование / план развития / корректирующие
  меры / административные меры). К1–К5 считаются автоматически из ежедневных
  проверок за период; Сона может **переопределить** любое значение вручную
  (одно заполнение → отчёт формируется сам).
- **Уведомления в Telegram** — отчёт аудитора, дневная и недельная сводки (cron на сервере).

## Стек

- **client** — React + Vite + TypeScript.
- **server** — Node + Express + TypeScript (через `tsx`), REST API, JWT-авторизация,
  cron-рассылка отчётов в Telegram.
- **БД** — Supabase (проект «OB FAQ»). Переиспользуются справочники
  `mqa_chats` (компании + связка с бухгалтером) и `mqa_accountants`.
- **Деплой** — Render (один Web Service: сервер раздаёт собранный клиент + API + cron).

## Модель данных (`sqa_*`)

| Таблица | Назначение |
|---|---|
| `sqa_reviews` | Ежедневная проверка по компании (тип отчёта, риск, чек-лист, оценка %, комментарий). Колонка `financials` — доходы/расходы (kind/section/amount/note), заполняется из формы |
| `sqa_efficiency_overrides` | Ручные правки Соны к взвешенному скорингу «Общая оценка» (К1–К5 за период; NULL = авто из проверок) |
| `sqa_accountant_workload` | «Всего отчётов» в работе у бухгалтера (знаменатель проверено/всего) |
| `sqa_daily_plan` | План на дату: сколько отчётов назначено каждому бухгалтеру |
| `sqa_tickets` | Авто-тикеты из проверок-«проблем» (приоритет + «ОЧЕНЬ СРОЧНО») |
| `sqa_criteria` | Критерии оценки (наполнение — TODO, согласовать с Соной/Лилит) |
| `sqa_notifications` | Лог отправок отчётов в Telegram |
| `sqa_users` | Учётные записи для входа в приложение |
| вьюхи `sqa_daily_report`, `sqa_weekly_report`, `sqa_accountant_daily`, `sqa_sona_efficiency` | Отчёты |

Справочники (read-only): `mqa_chats`, `mqa_accountants`.

## Локальный запуск

```bash
cp .env.example .env   # заполнить значения
npm install            # ставит зависимости корня
npm run install:all    # ставит зависимости client и server
npm run dev            # client (Vite) + server параллельно
```

Версия Node — **22** (см. `.nvmrc` и `render.yaml`). Нужна Node ≥ 22:
`@supabase/supabase-js` использует нативный WebSocket, которого нет в Node 20.

### Проверки перед коммитом

```bash
npm run typecheck   # tsc --noEmit для server и client
npm test            # юнит-тесты сервера (scoring, Итог Q, финансы, форматы)
npm run check       # typecheck + тесты + сборка клиента (то же, что гоняет CI)
```

То же самое автоматически выполняет **GitHub Actions** (`.github/workflows/ci.yml`)
на каждый push и PR в `main` — типизация обоих пакетов и сборка клиента.
В сессиях **Claude Code on the web** зависимости ставит хук
`.claude/hooks/session-start.sh`.

## Деплой на Render

Render использует `render.yaml`. Переменные окружения задаются в дашборде Render
(секреты не хранятся в репозитории):

- `SUPABASE_URL`, `SUPABASE_SERVICE_KEY`
- `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`
- `JWT_SECRET`
- `SQA_ADMIN_EMAIL`, `SQA_ADMIN_PASSWORD` (для создания первой учётки при старте)
- `TZ` (по умолчанию `Asia/Yerevan`)

## Открытые вопросы / TODO

- **Критерии и шкала** качества бухучёта — согласовать с Соной / Лилит / Эмилией,
  затем засидить в `sqa_criteria`.
- **Оценка % и «Баллы»** — формула чек-листа (9 пунктов, доля «хороших» × 100)
  и банды баллов взяты из рабочего файла Соны (`server/src/efficiency.ts`);
  при необходимости подправить веса/банды.
- **«Проверено / всего»** — «проверено» = число проверок за **выбранный день**
  (подтверждено образцом Соны: «Наира З. 12/63» за 15/06 = ровно 12 строк за день),
  «всего» = норма на бухгалтера, Сона ведёт вручную в отчёте (`sqa_accountant_workload`).
- **Журнал «Общая оценка» (взвешенный Итог Q)** — реализован на вкладке
  «Общая оценка». К1–К5 пока **выводятся** из 9-пунктного чек-листа
  (ошибки/штрафы→К1, просрочка→К2, корректность+формат→К3, подписи+первичка+
  стандарты→К4, запись-«проблема»→К5); при необходимости уточнить маппинг
  с Соной. Веса (0.1/0.3/0.2/0.3/0.1) и уровни сверены с её файлом
  (Итог Q совпадает: 84/85/94/86).
- **Доходы/расходы** — форма пишет их в `sqa_reviews.financials`; разбивка по
  компаниям и разделам видна в «Отчётах». При необходимости — справочник
  типовых разделов.
- **Значение «ОК»** в уведомлениях — уточнить (отдельный канал/мессенджер?).
